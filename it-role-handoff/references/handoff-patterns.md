# Handoff Patterns

This file defines common role-to-role handoff patterns inside the target organizational structure.

## General rule

A handoff pattern describes:
- the source role,
- the kind of output that role usually produces,
- the role that most naturally receives the next handoff,
- and the reason for that transition.

These are patterns, not rigid laws. Use them to keep work realistic and organized.

## Business Analyst

### Typical outputs
- business requirements
- functional breakdowns
- scope clarification
- assumptions and open questions

### Common next handoffs
- Web Developer
- Mobile Developer
- UI/UX Designer
- Product & Project Manager
- Quality Assurance

### Why
Business Analyst work usually becomes the basis for design, implementation, planning, and validation.

## UI/UX Designer

### Typical outputs
- interface flow
- layout direction
- interaction guidance
- usability recommendations

### Common next handoffs
- Web Developer
- Mobile Developer
- Graphic Designer
- Product & Project Manager

### Why
UI/UX outputs often need to be implemented by developers or supported by visual design and planning roles.

## Graphic Designer

## Practical use rule

When the user asks for a role-based response:
1. complete the requested role's part first,
2. identify the most natural next role if a handoff is needed,
3. explain the handoff briefly,
4. avoid forcing a handoff when the task can reasonably stay within one role.
